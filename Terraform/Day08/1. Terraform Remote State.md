## Terraform Remote State with S3 Backend: With and Without DynamoDB Locking

## Introduction
This document provides a structured overview of the Terraform Remote State configuration. It includes the code from `main.tf`, `terraform.tf`, and `variables.tf`, explains the differences between using DynamoDB for state locking and not using it, and demonstrates how to initialize Terraform with different environments (dev, prod, stage) using `terraform init --reconfigure` to manage separate tfstate files.

### What is Terraform Remote State?

By default, Terraform stores state locally in a file called `terraform.tfstate`. This file keeps track of all resources Terraform manages. In a team or production environment, storing state locally can cause issues like conflicts, overwrites, or outdated changes.

**Remote state** allows storing the state file in a shared backend like AWS S3. It provides centralized, consistent, and reliable state management.

### Why Remote State is Important

* Shared access to the latest state file
* Enables team collaboration
* Centralized backup with versioning
* Prevents state file corruption
* Supports Terraform modules and automation pipelines

### Why Use DynamoDB?

When multiple people use Terraform together, locking is required. DynamoDB provides a lock mechanism so that only one person can modify infrastructure at a time. It avoids race conditions or duplicate resource creation.

### Components Required

1. **S3 Bucket** - Stores the state file
2. **DynamoDB Table** - Used for state locking
3. **Terraform backend block** - Configures remote backend

### Step-by-Step Setup

#### Step 1: Create an S3 Bucket

```bash
aws s3api create-bucket \
  --bucket terraform-tfstate-file-storage-2002 \
  --region ap-south-1 \
  --create-bucket-configuration LocationConstraint=ap-south-1

aws s3api put-bucket-versioning \
  --bucket terraform-tfstate-file-storage-2002 \
  --versioning-configuration Status=Enabled
```

#### Step 2: Create DynamoDB Table for Locking

```bash
aws dynamodb create-table \
  --table-name tfstate-file-locks \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

#### Step 3: Terraform Backend Configuration

File: `terraform.tf`

```hcl
terraform {
  required_version = ">= 1.3"

  backend "s3" {
    bucket         = "terraform-tfstate-file-storage-2002"
    key            = "dev/terraform.tfstate"
    region         = "ap-south-1"
    dynamodb_table = "tfstate-file-locks"
    encrypt        = true
  }
}
```

Note: This file should not use variables and must be initialized before running any plan or apply.

### terraform.tf
This file configures the Terraform backend and required providers. Without backend configuration, the tfstate file is stored locally. With backend, it's stored remotely in S3.

The current setup uses S3 backend for dev environment without DynamoDB locking.

```hcl
terraform {

    required_version = ">= 0.14"
    required_providers {
        aws = {
            source  = "hashicorp/aws"
            version = "~> 4.0"
        }
    }

    backend "s3" {
        bucket = "terraform-tfstate-file-storage-2002"
        key    = "dev/terraform.tfstate"
        region = "ap-south-1"
        encrypt = true
        use_lockfile = false
    }
}
```

## State Locking Comparison

**Key Points:**
- **Backend Storage**: Without backend config, tfstate is stored locally. With S3 backend, it's remote.
- **With DynamoDB Locking**: Only one user can modify at a time; others get errors until lock releases. Prevents conflicts and ensures consistency.
- **Without Locking**: Multiple users can modify simultaneously, risking conflicts and inconsistencies. Requires team coordination.

| Aspect | With DynamoDB Locking | Without DynamoDB Locking |
|--------|-----------------------|--------------------------|
| **State Locking** | Uses DynamoDB table to lock state file | No locking mechanism |
| **Concurrency Control** | Only one user at a time | Multiple users simultaneously |
| **Risks** | Prevents conflicts | Potential conflicts and inconsistencies |
| **Coordination** | Automatic via locking | Manual communication required |
| **Configuration** | Includes `dynamodb_table` | Uses `use_lockfile = false` or omits |

## Backend Configuration Examples

### With DynamoDB Locking
```hcl
backend "s3" {
    bucket         = "terraform-tfstate-file-storage-2002" #S3 bucket name where tfstate file will be stored
    key            = "prod/terraform.tfstate" #Path within the bucket where tfstate file will be stored
    region         = "ap-south-1"
    dynamodb_table = "tfstate-file-locks" #DynamoDB table name for state locking
    encrypt        = true
}
```

### Without DynamoDB Locking
```hcl
backend "s3" {
    bucket      = "terraform-tfstate-file-storage-2002"
    key         = "dev/terraform.tfstate"
    region      = "ap-south-1"
    encrypt     = true
    use_lockfile = false 
}
```
`use_lockfile = false` This option disables the use of lock files for state locking. 
When set to false, Terraform will not create lock files in the S3 bucket to prevent concurrent modifications to the state file. This can be useful in scenarios where multiple users need to access and modify the state file simultaneously, but it also increases the risk of conflicts and inconsistencies in the infrastructure. It is important to have a clear communication and coordination among team members when using this option to avoid conflicts and ensure that the infrastructure remains in a consistent state.
### Dev Environment
```bash
terraform init --reconfigure -backend-config="key=dev/terraform.tfstate"
```

### Prod Environment
```bash
terraform init --reconfigure -backend-config="key=prod/terraform.tfstate"
```

### Stage Environment
```bash
terraform init --reconfigure -backend-config="key=stage/terraform.tfstate"
```

**Note:** The backend in `terraform.tf` is set for dev. Adjust or use `-backend-config` for other environments.

## Additional Notes
- Remote state storage uses S3 with encryption enabled.
- EC2 instance is tagged for dev environment.
- Ensure S3 bucket and DynamoDB table (if locking) exist before init.
- For production, consider enabling DynamoDB locking for safety.


#### Initialize and Apply

```bash
terraform init
terraform apply
```

### How to Verify

* S3 bucket will contain the file `dev/terraform.tfstate`
* DynamoDB table will show lock entry when `apply` is running

### Best Practices

* Use a different key per environment (e.g., dev/terraform.tfstate, prod/terraform.tfstate)
* Enable versioning in the S3 bucket
* Apply IAM policies to restrict delete actions
* Keep backend block separate and clean
* Never manually edit `.tfstate` file
