## Variables in Terraform

Variables allow you to supply values to your configuration **without modifying the main Terraform code**. Think of them as function parameters — they let the same configuration behave differently depending on what values you pass in.

### Why Use Variables?

| Benefit | What It Means in Practice |
|---|---|
| **Reusability** | One config file works across dev, test, and prod — just swap the values |
| **Clean Code** | No hardcoded IDs or region names scattered across files |
| **Flexible** | Values can be changed at runtime via CLI, `.tfvars` files, or env vars |
| **Consistent** | Centralized variables enforce uniform settings across all resources |
| **Secure** | Mark sensitive values with `sensitive = true` to hide them from logs |

---

### Variable Types

| Type | Description | Example |
|---|---|---|
| `string` | Text value | `"dev"`, `"us-east-1"` |
| `number` | Numeric value | `10`, `20`, `400` |
| `bool` | True or False | `true`, `false` |
| `list` | Ordered collection of values | `["t2.micro", "t3.micro"]` |
| `map` | Key-value pairs | `{ env = "dev", tier = 1 }` |
| `object` | Group of named attributes with defined types | `{ name = "dev", size = 2 }` |
| `tuple` | Ordered list with **mixed** data types | `["dev", 2, true]` |

---

### Declaring and Using Variables

```hcl
# variables.tf — Declare your variables here
variable "instance_type" {
  description = "The EC2 instance type to use"
  type        = string
  default     = "t3.micro"
}

variable "environment" {
  description = "Deployment environment"
  type        = string
  # No default — must be supplied at runtime
}

variable "allowed_ports" {
  description = "List of ports to open in the security group"
  type        = list(number)
  default     = [80, 443, 22]
}

variable "db_password" {
  description = "Database password"
  type        = string
  sensitive   = true   # Value will be hidden in logs and output
}
```

```hcl
# main.tf — Reference variables using var.<name>
resource "aws_instance" "web" {
  ami           = "ami-02d05f76acbed4e3e"
  instance_type = var.instance_type   # References the variable above

  tags = {
    Environment = var.environment
  }
}
```
### Providing Variable Values
There are multiple ways to pass values — listed in order of precedence (highest wins):
```bash
# 1. Command-line flag (highest precedence)
terraform apply -var="instance_type=t3.medium"

# 2. .tfvars file (auto-loaded if named terraform.tfvars)
terraform apply -var-file="production.tfvars"

# 3. Environment variable
export TF_VAR_instance_type="t3.medium"
```

> 💡 **Interview Tip:** The interviewer may ask *"How do you pass variable values?"* — there are three ways: via CLI (`-var`), via a `.tfvars` file (`-var-file`), or via environment variables prefixed with `TF_VAR_` (e.g., `TF_VAR_instance_type=t3.micro`).

> 💡 **Interview Tip:** Know the variable precedence order — `TF_VAR_*` env vars < `terraform.tfvars` < `-var-file` flag < `-var` flag.

### Variable Validation
```bash
variable "environment" {
  type        = string
  description = "Deployment environment"

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}
```

## `.tfvars` Files and Workspaces

### `.tfvars` Files — Separating Config from Logic

A `.tfvars` file is where you **assign values** to the variables you declared. It separates the *what* (variable declarations) from the *which values* (environment-specific assignments), keeping your code reusable and clean.

```hcl
# dev.tfvars
instance_type = "t2.micro"
environment   = "dev"
allowed_ports = [80, 22]
```

```hcl
# prod.tfvars
instance_type = "t3.large"
environment   = "prod"
allowed_ports = [80, 443]
```

You apply them like this:

```bash
terraform apply -var-file="dev.tfvars"
terraform apply -var-file="prod.tfvars"
```

### The Problem with `.tfvars` Alone ⚠️

Here's the critical limitation — even though your variable *values* are different per environment, **all environments share the same state file**. This means applying `prod.tfvars` after `dev.tfvars` will cause Terraform to see the dev resources as "outdated" and **destroy them**. That is not production-grade behavior.

```
dev.tfvars  ──┐
test.tfvars ──┼──▶  Same terraform.tfstate  ──▶  ❌ Conflict & destruction risk
prod.tfvars ──┘
```

This is exactly the problem that **Terraform Workspaces** solve.

---

### Terraform Workspaces — Isolated State per Environment

Workspaces give each environment its **own state file**, so dev, test, and prod never interfere with each other.

```
dev workspace   ──▶  terraform.tfstate.d/dev/terraform.tfstate
test workspace  ──▶  terraform.tfstate.d/test/terraform.tfstate
prod workspace  ──▶  terraform.tfstate.d/prod/terraform.tfstate
```

#### Workspace Commands

```bash
# See all existing workspaces (default workspace exists always)
terraform workspace list

# Create a new workspace and switch to it immediately
terraform workspace new dev

# Switch between workspaces
terraform workspace select prod

# Check which workspace you're currently in
terraform workspace show
```

#### Using Workspace Name Dynamically in Code

```hcl
# You can reference the current workspace name inside your config
resource "aws_instance" "web" {
  ami           = "ami-02d05f76acbed4e3e"
  instance_type = terraform.workspace == "prod" ? "t3.large" : "t2.micro"

  tags = {
    Environment = terraform.workspace   # Automatically uses "dev", "test", or "prod"
  }
}
```

#### Cleaning Up Workspaces

Deleting a workspace has two strict rules:

1. **The workspace must be empty** — all resources must be destroyed, removed, or migrated first.
2. **You cannot delete the workspace you're currently in** — switch to a different one first.

```bash
# Example: deleting the "dev" workspace safely
terraform workspace select prod    # Step 1: switch away from dev
terraform workspace delete dev     # Step 2: now safe to delete
```

> 💡 **Interview Tip:** Workspaces are great for isolation, but many teams prefer **separate directories or separate backend configurations** per environment for even stricter separation. Knowing both approaches shows maturity.

## Output Blocks

An **output block** exposes values from your infrastructure after `terraform apply` runs — think of it as Terraform's way of printing useful information like IPs, ARNs, or DNS names to the console (or to other Terraform configurations).

```hcl
# outputs.tf
output "public_ip" {
  description = "The public IP of the web server"
  value       = aws_instance.web.public_ip
}

output "instance_id" {
  description = "The EC2 instance ID"
  value       = aws_instance.web.id
}

output "db_password" {
  description = "Database password"
  value       = aws_db_instance.mydb.password
  sensitive   = true   # Will be hidden in terminal output
}
```
> 💡 **Interview Tip:** `sensitive = true` masks the value in terminal output but it is **still stored in plain text in the state file**. Use encrypted remote backends (like S3 + SSE) for true security.

#### Output Commands

```bash
# Display all output values
terraform output

# Display a specific output value
terraform output public_ip

# Display all outputs in JSON format (useful for scripting)
terraform output -json
```

> 💡 **Interview Tip:** Outputs are also used to **share data between modules** — a child module's output can be consumed by the parent module using `module.<name>.<output_name>`. This is a very common interview topic.

## Data Source Blocks

A **data block** fetches information about infrastructure that **already exists** — outside your current Terraform configuration. Instead of hardcoding an AMI ID or VPC ID, you look it up dynamically.

Think of data blocks as **read-only queries** to your cloud provider.

```hcl
# Fetch the latest Amazon Linux 2 AMI dynamically — no hardcoding needed
data "aws_ami" "latest_amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["amzn2-ami-hvm-*-x86_64-gp2"]
  }
}

# Use the fetched AMI in your resource
resource "aws_instance" "web" {
  ami           = data.aws_ami.latest_amazon_linux.id   # Dynamically resolved
  instance_type = "t3.micro"
}
```

```hcl
# Fetch an existing VPC by its tag — no need to hardcode the VPC ID
data "aws_vpc" "main" {
  filter {
    name   = "tag:Name"
    values = ["main-vpc"]
  }
}

# Use the fetched VPC ID in a subnet resource
resource "aws_subnet" "public" {
  vpc_id     = data.aws_vpc.main.id
  cidr_block = "10.0.1.0/24"
}
```

### Key Uses of Data Blocks

Data blocks help you avoid hardcoding by looking up existing infrastructure dynamically — such as fetching the latest AMI, referencing a VPC managed by another team, or reading outputs from a separate Terraform state using `terraform_remote_state`.

> 💡 **Interview Tip:** A common question is *"What's the difference between a resource block and a data block?"* — a `resource` block **creates and manages** infrastructure, while a `data` block only **reads** existing infrastructure. Data blocks never create anything.

## Dynamic Blocks

A **dynamic block** generates repeating nested blocks inside a resource automatically, using a loop. This is Terraform's answer to the *"I don't know in advance how many sub-blocks I need"* problem.

### The Problem Without Dynamic Blocks

```hcl
# ❌ Repetitive and not scalable — hardcoded ingress rules
resource "aws_security_group" "web_sg" {
  name = "web-sg"

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

### The Solution With Dynamic Blocks

```hcl
# variables.tf — Define the ports as a variable
variable "ingress_ports" {
  description = "List of ports to open"
  type        = list(number)
  default     = [80, 443, 22]
}
```

```hcl
# main.tf — Use a dynamic block to generate ingress rules automatically
resource "aws_security_group" "web_sg" {
  name = "web-sg"

  # 'dynamic' keyword + the block type you want to repeat ("ingress")
  dynamic "ingress" {
    for_each = var.ingress_ports   # Iterate over each port in the list

    content {
      # 'ingress.value' refers to the current element in the loop
      from_port   = ingress.value
      to_port     = ingress.value
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
    }
  }
}
```

### Dynamic Block with a Map (More Control)

```hcl
# Use a map to carry more information per rule
variable "ingress_rules" {
  type = map(object({
    port     = number
    protocol = string
  }))
  default = {
    http  = { port = 80,  protocol = "tcp" }
    https = { port = 443, protocol = "tcp" }
    ssh   = { port = 22,  protocol = "tcp" }
  }
}

resource "aws_security_group" "web_sg" {
  name = "web-sg"

  dynamic "ingress" {
    for_each = var.ingress_rules   # Iterate over the map

    content {
      from_port   = ingress.value.port       # Access map values
      to_port     = ingress.value.port
      protocol    = ingress.value.protocol
      cidr_blocks = ["0.0.0.0/0"]
    }
  }
}
```

> 💡 **Interview Tip:** Dynamic blocks are often asked alongside `for_each` and `count`. The key distinction is that `for_each`/`count` creates **multiple resource instances**, while `dynamic` creates **multiple nested blocks within a single resource**.
